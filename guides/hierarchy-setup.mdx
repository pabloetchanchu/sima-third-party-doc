## Overview

SIMA organizes agricultural data in a strict hierarchy. Every Work Order ultimately ties to a **cultivation**, which is the combination of a plot, a campaign, and a crop. You need to set up this hierarchy before any field operations can be registered.

```
Society (Farmer)
  └── Establishment (Farm/Field)
        └── Plot (Lote)
              └── Master Campaign → Campaign
                    └── Cultivation (Plot + Campaign + Crop)
                          └── Work Order
```

<Info>
  If your system already has establishments and plots, create them in SIMA and link them using the `external_code` field to map between your IDs and SIMA's `local_id`.
</Info>

---

## Setup order

The order matters — each level depends on the one above it.

<Steps>
  <Step title="Verify your Society (Farmer)">
    Your society context is returned in the login response as `society.id`. This is the root of all your data. You don't create societies via API — they are provisioned by SIMA during onboarding.
  </Step>
  <Step title="Create Establishments">
    An establishment is a farm or field group belonging to a farmer. Requires a `farmer_id`.
  </Step>
  <Step title="Create Plots">
    A plot (lote) is a specific parcel of land within an establishment. Requires an `establishment_id`.
  </Step>
  <Step title="Create Master Campaigns and Campaigns">
    A master campaign is a crop season (e.g. "2024/25"). A campaign is a specific crop within that season on a set of plots.
  </Step>
  <Step title="Create Cultivations">
    A cultivation links a plot to a campaign and a specific crop. This is the entity referenced in every Work Order line.
  </Step>
</Steps>

## Establishments

### Create

```
POST /api/v3/third_party/establishments
```

```json
{
  "farmer_id": 13873,
  "name": "La Correntina",
  "observations": "Northern sector, irrigated"
}
```

**Response:**

```json
{
  "id": 118422,
  "farmer_id": 13873,
  "name": "La Correntina",
  "observations": "Northern sector, irrigated",
  "location": null,
  "created_at": "2024-08-30T18:03:13-03:00",
  "active": true
}
```

Store the returned `id` — you'll need it to create plots.

### Update

```
PATCH /api/v3/third_party/establishments/{id}
```

### Delete

```
DELETE /api/v3/third_party/establishments/{id}
```

### List

```
GET /api/v3/third_party/establishments
```

**Key filters:**

| Parameter | Type | Description |
|---|---|---|
| `farmer_id` | int | Filter by farmer |
| `updated_at_from` | datetime | Detect new or changed establishments |
| `page` / `size` | int | Pagination |

## Plots

### Create

```
POST /api/v3/third_party/plots
```

```json
{
  "establishment_id": 118422,
  "name": "Lote 1 Norte",
  "area": 150.5
}
```

### List

```
GET /api/v3/third_party/plots
```

Plots can be filtered by `establishment_id` and `updated_at_from` for change detection.

## Master Campaigns and Campaigns

A **Master Campaign** represents a crop season (e.g. "2024/25 Fine Grains"). A **Campaign** is the specific crop + period tied to a set of plots within that season.

### List Master Campaigns

```
GET /api/v3/third_party/master_campaigns
```

### List Campaigns

```
GET /api/v3/third_party/campaigns
```

**Key filters:**

| Parameter | Type | Description |
|---|---|---|
| `establishment_id` | int | Campaigns within an establishment |
| `updated_at_from` | datetime | Change detection |
| `active` | bool | Filter active campaigns only |

---

## Cultivations

A cultivation links a **plot + campaign + crop**. Every supply and labour line in a Work Order references a `cultivation_id`.

### List Cultivations

```
GET /api/v3/third_party/cultivations
```

This is the endpoint you'll query most often — it returns the IDs you need for Work Order creation.

**Typical query to get cultivations for a specific establishment:**

```bash
GET /api/v3/third_party/cultivations?establishment_id=118422&active=true
```

## Linking your IDs to SIMA IDs

When your system has its own identifiers, use `external_code` to create a persistent mapping:

```json
{
  "farmer_id": 13873,
  "name": "La Correntina",
  "external_code": "YOUR-ERP-ESTAB-001"
}
```

You can then look up the entity by your own code: `GET /api/v3/third_party/establishments?external_code=YOUR-ERP-ESTAB-001`

---

## Handling existing entities

If your client already has establishments and plots configured in SIMA (e.g. set up manually via the web app), don't recreate them. Instead:

1. List existing establishments: `GET /api/v3/third_party/establishments`
2. Match by name or location to your own records
3. Store SIMA's `local_id` mapped to your own ID
4. Use `external_code` on future creates/updates to maintain the link
